Luồng Request-Response Chi tiết

Ví dụ: Tạo một Task mới trong hệ thống

1
User nhập liệu trên Form
Người dùng điền form tạo task: tiêu đề, mô tả, người thực hiện, ưu tiên, KPI weight, và các chỉ số kỳ vọng (volume, quality, difficulty, ahead-of-schedule).
React Component useState Form Validation
2
Frontend gọi Mutation
Custom hook useCreateTask() gọi useMutation() của TanStack Query, gửi POST request đến /api/tasks với body JSON chứa dữ liệu task.
useMutation fetch() JSON body
3
Middleware kiểm tra Auth
Request đi qua middleware Next.js → kiểm tra JWT cookie → validate session với Supabase → nếu hợp lệ cho phép đi tiếp, nếu không redirect về /login.
middleware.ts JWT Cookie updateSession()
4
API Route xử lý
① Auth: getAuthProfile() lấy user profile từ Supabase
② Authorization: requireMinRole(profile, "team_leader") kiểm tra quyền
③ Validation: Kiểm tra assignee tồn tại, cùng org, trong phạm vi quản lý
④ Calculate: Tính expect_score = 0.4×vol + 0.3×qual + 0.2×diff + 0.1×ahead
getAuthProfile() requireMinRole() Input validation
5
Database INSERT + RLS
Supabase client thực hiện INSERT vào bảng tasks. Row-Level Security (RLS) tự động kiểm tra: org_id = user_org_id() và role phù hợp. PostgreSQL triggers ghi log vào audit_logs và task_status_logs.
PostgreSQL INSERT RLS Policy DB Triggers
6
API trả Response
API trả về JSON response với status 201 (Created), chứa task ID và toàn bộ dữ liệu task đã tạo. Frontend nhận response → onSuccess callback kích hoạt.
jsonResponse() 201 Created
7
Cache Invalidation & Refetch
queryClient.invalidateQueries(taskKeys.all) → Tất cả danh sách task tự động refetch → React Query merge cache → Component re-render với data mới.
invalidateQueries Auto refetch Cache merge
8
Realtime Broadcast
PostgreSQL trigger phát sự kiện postgres_changes → Supabase Realtime broadcast qua WebSocket → Tất cả client đang subscribe nhận được → Auto-invalidate query → Gửi notification cho assignee → Cập nhật unread count.
postgres_changes WebSocket Notification trigger

Security Layers (Các lớp bảo mật)

Layer 1: Supabase Authentication (JWT)
User đăng nhập → Supabase Auth cấp JWT + refresh token → lưu trong httpOnly cookies (không accessible từ JS). Mỗi request đều mang JWT để xác thực danh tính.
Layer 2: Next.js Middleware
Middleware chặn mọi request trước khi vào API routes. Kiểm tra session hợp lệ, refresh token nếu cần. Whitelist: /login, /register, /api/auth, /api/health, /api/cron, /api/webhooks. Tất cả route khác phải authenticated.
Layer 3: Role-Based Access Control (RBAC)
API routes kiểm tra quyền qua requireMinRole(profile, role). Phân cấp: admin > leader > director > head > team_leader > staff. Mỗi endpoint yêu cầu role tối thiểu khác nhau. Permission utilities: hasMinRole(), canCreateProject().
Layer 4: Row-Level Security (RLS)
PostgreSQL RLS policies tự động lọc data theo org_id. Users chỉ thấy data của tổ chức mình. Team leaders thấy members của team. Heads thấy phòng ban. Directors/Leaders thấy rộng hơn. Đảm bảo data isolation ở tầng database.
Layer 5: Secret-Protected Routes
Cron endpoints bảo vệ bằng CRON_SECRET header — chỉ Vercel/scheduler gọi được. Webhook endpoints yêu cầu WEBHOOK_SECRET. Không ai từ bên ngoài có thể trigger các job nhạy cảm.
Layer 6: Audit Trail
Mọi hành động quan trọng được ghi vào audit_logs. Task status changes ghi vào task_status_logs. Database triggers tự động capture thay đổi — không bypass được. Phục vụ compliance và truy vết.
